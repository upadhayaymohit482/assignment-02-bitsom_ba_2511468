## Database Recommendation

For a healthcare startup building a patient management system, I would recommend MySQL as the primary database for the core application. Patient management involves highly sensitive and transactional workflows such as registration, appointments, prescriptions, diagnoses, billing, and updates to clinical history. These operations require strong ACID guarantees: atomicity so partial transactions do not occur, consistency so records remain valid, isolation so concurrent users do not corrupt each other’s work, and durability so committed data is not lost. In a hospital setting, eventual consistency is usually not acceptable for the system of record.

MongoDB is valuable when the data is flexible or document-shaped, but for the main patient system its BASE-style design and more relaxed consistency model make it less suitable than a relational engine. CAP theorem reasoning also supports this choice: for healthcare records, consistency should be prioritized over availability during partitions, because inaccurate clinical data can directly affect patient safety.

My answer would change somewhat if the startup also needed a fraud detection module. Fraud detection often consumes large, high-volume, semi-structured data streams such as logs, claims histories, device events, and behavioral signals. That workload benefits from a more flexible and horizontally scalable store, and MongoDB could fit well there. Even then, I would not replace MySQL for the patient management core. Instead, I would recommend a hybrid architecture: MySQL for the transactional patient system, and MongoDB or another scalable analytical store for fraud-related features, logs, and model-serving inputs. That design keeps the clinical system strongly consistent while still supporting fast experimentation and flexible analytics for fraud detection.
