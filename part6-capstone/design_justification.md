## Storage Systems

I would use four different storage systems because the hospital has four clearly different workloads. For the operational system of record, patient registrations, encounters, diagnoses, treatment history, and billing transactions should live in a relational OLTP database such as PostgreSQL. That database gives strong ACID guarantees, which are essential for clinical correctness. For monthly management reporting and readmission-risk modeling, I would replicate curated historical data into a data warehouse such as BigQuery, Snowflake, or Redshift. Warehouses are better for long scans, aggregation, and feature generation than transactional systems. For doctors’ plain-English search over patient history, I would add a vector database such as pgvector, Pinecone, or Weaviate. Clinical notes, discharge summaries, and past event narratives can be chunked, embedded, and indexed there for semantic retrieval. Finally, real-time ICU vitals should be written to a time-series database such as InfluxDB or TimescaleDB because it is optimized for high-frequency timestamped measurements and fast recent-window queries.

## OLTP vs OLAP Boundary

In this design, the OLTP boundary ends at the point where data is being used to run day-to-day hospital operations. The PostgreSQL system handles live patient updates, billing entries, medication orders, and other transaction-heavy workflows. Once data leaves that operational path through ETL or streaming replication into curated analytical storage, it crosses into OLAP territory. The warehouse becomes the source for historical analysis, KPI dashboards, departmental cost reports, bed-occupancy summaries, and readmission-risk model training. The vector-search layer is not the transactional source of truth either; it is a retrieval system built from copies of patient notes and history. Similarly, the time-series store handles ICU telemetry efficiently, but reporting-grade rollups from that telemetry should flow into the analytical layer when needed.

## Trade-offs

The biggest trade-off is architectural complexity. Using separate systems for OLTP, analytics, vector retrieval, and streaming telemetry gives the right performance characteristics for each workload, but it also increases operational overhead: more pipelines, more monitoring, more schema governance, and more failure points. I would mitigate that in three ways. First, I would standardize ingestion with a central orchestration layer such as Airflow plus event streaming via Kafka, so data movement is observable and repeatable. Second, I would define clear ownership and contracts for each dataset, especially between the transactional schema and the warehouse or vector index. Third, I would start with the minimum viable set of tools—for example PostgreSQL + warehouse + pgvector + TimescaleDB—before expanding to separate managed services only when scale demands it. That keeps the design powerful without letting platform complexity overwhelm the hospital team.
